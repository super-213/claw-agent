## 架构分析报告

### 当前架构的优点
- 简洁直观的单文件实现
- 清晰的技能加载机制
- 基于标记的输出解析（[命令]、[完成]）

### 架构问题与改进方向

#### 1. **关注点分离不足**
当前所有逻辑混在 `main()` 函数中，包括：
- 配置管理
- 消息处理
- 技能加载
- 命令执行
- 对话循环

**建议**：采用分层架构，将不同职责分离到独立的类/模块

#### 2. **缺乏扩展性**
- 硬编码的标记解析（[命令]、[完成]）
- 技能系统缺乏插件化机制
- 无法动态注册新的输出处理器

**建议**：引入策略模式和责任链模式

#### 3. **错误处理薄弱**
- 空的 except 块会隐藏错误
- 缺乏日志系统
- 命令执行无超时保护

**建议**：完善异常处理和日志系统

#### 4. **测试困难**
- 紧耦合的设计难以单元测试
- 依赖外部 API 和文件系统

**建议**：依赖注入和接口抽象

#### 5. **配置管理简陋**
- 硬编码的 API Key（安全风险）
- 缺乏环境变量支持
- 无配置验证

**建议**：使用配置文件 + 环境变量

## 🏗️ 架构重构方案

### 核心问题识别

1. **单体函数设计** - 所有逻辑集中在 `main()` 中，违反单一职责原则
2. **硬编码依赖** - API Key、文件路径、标记格式都硬编码
3. **缺乏抽象层** - 技能、命令执行、消息处理没有统一接口
4. **状态管理混乱** - `active_skill` 变量在循环中手动管理
5. **扩展性差** - 添加新功能需要修改核心逻辑

### 关键重构点

#### 1️⃣ **配置管理重构**
```python
# 当前问题：硬编码 API Key
CONFIG = {"api_key": "sk-xxx..."}

# 重构方案：
class ConfigManager:
    def __init__(self):
        self.config = self._load_from_env()
        self._validate()
    
    def _load_from_env(self):
        return {
            "api_key": os.getenv("DASHSCOPE_API_KEY"),
            "base_url": os.getenv("API_BASE_URL", "https://..."),
            # 支持 .env 文件
        }
```

#### 2️⃣ **技能系统插件化**
```python
# 当前问题：硬编码技能加载逻辑
def load_skill(name):
    path = Path("skills") / name / f"{name}.md"
    return path.read_text()

# 重构方案：
class SkillRegistry:
    def __init__(self):
        self._skills = {}
        self._discover_skills()
    
    def register(self, skill: BaseSkill):
        self._skills[skill.name] = skill
    
    def get(self, name: str) -> Optional[BaseSkill]:
        return self._skills.get(name)

class BaseSkill(ABC):
    @abstractmethod
    def load_context(self) -> str:
        pass
    
    @abstractmethod
    def parse_output(self, text: str) -> Optional[dict]:
        pass
```

#### 3️⃣ **响应处理责任链**
```python
# 当前问题：if-elif 链式判断
if "[完成]" in reply: ...
elif "[命令]" in reply: ...
elif active_skill: ...

# 重构方案：
class ResponseHandler(ABC):
    def __init__(self, next_handler=None):
        self._next = next_handler
    
    def handle(self, response: str, context: Context) -> HandlerResult:
        if self.can_handle(response):
            return self.process(response, context)
        elif self._next:
            return self._next.handle(response, context)
        return HandlerResult.CONTINUE

class CompletionHandler(ResponseHandler):
    def can_handle(self, response: str) -> bool:
        return "[完成]" in response

class CommandHandler(ResponseHandler):
    def can_handle(self, response: str) -> bool:
        return "[命令]" in response
```

#### 4️⃣ **命令执行安全增强**
```python
# 当前问题：无超时、无安全检查
result = subprocess.run(cmd, shell=True, ...)

# 重构方案：
class CommandExecutor:
    DANGEROUS_PATTERNS = [r'rm\s+-rf\s+/', r'mkfs', ...]
    
    def execute(self, cmd: str, timeout: int = 30) -> ExecutionResult:
        if self._is_dangerous(cmd):
            raise SecurityError(f"危险命令: {cmd}")
        
        try:
            result = subprocess.run(
                cmd, shell=True, 
                capture_output=True, 
                text=True, 
                timeout=timeout
            )
            return ExecutionResult(result.stdout, result.returncode)
        except subprocess.TimeoutExpired:
            return ExecutionResult.timeout()
```

#### 5️⃣ **对话管理解耦**
```python
# 当前问题：messages 列表在 main() 中手动管理
messages = [...]
messages.append(...)

# 重构方案：
class ConversationManager:
    def __init__(self, system_prompt: str):
        self._messages = [{"role": "system", "content": system_prompt}]
        self._history = []
    
    def add_user_message(self, content: str):
        self._messages.append({"role": "user", "content": content})
    
    def add_assistant_message(self, content: str):
        self._messages.append({"role": "assistant", "content": content})
    
    def inject_skill_context(self, skill_name: str, content: str):
        self._messages.append({
            "role": "system",
            "content": f"## 激活技能：{skill_name}\n{content}"
        })
    
    def get_messages(self) -> list:
        return self._messages.copy()
```

### 重构收益

✅ **可维护性**：每个模块职责清晰，修改影响范围小  
✅ **可测试性**：依赖注入，可 mock 外部服务  
✅ **可扩展性**：新增技能/处理器无需修改核心代码  
✅ **安全性**：命令执行有白名单/黑名单机制  
✅ **可观测性**：统一日志系统，便于调试  
✅ **可配置性**：环境变量 + 配置文件分离
